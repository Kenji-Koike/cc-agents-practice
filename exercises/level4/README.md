# Level 4: 実践 - コードレビューエージェント

## 目標

- 実用的なコードレビューエージェントを設計・構築する
- プロンプトエンジニアリングでエージェントの品質を高める
- エージェントの出力品質を反復改善する

---

## Step 4-1: レビュー対象のコードを用意する

```
exercises/level4/src/ ディレクトリに以下のファイルを作成してください：

1. auth.py
   - パスワードを平文でログに出力している（セキュリティバグ）
   - JWT トークンの有効期限チェックがない
   - SQL クエリを文字列結合で作っている（SQLインジェクション）

2. api_client.py
   - タイムアウトなしの requests.get() を使用
   - エラーハンドリングなし
   - APIキーをコード内にハードコード

3. data_processor.py
   - 大きなファイルを一度に全部メモリに読み込む
   - N+1クエリパターン
   - 適切なインデックスなしのリスト検索
```

---

## Step 4-2: 基本的なレビューエージェントを作成する（v1）

まず素朴なバージョンを作ります。

```
.claude/agents/code-reviewer-v1.md を作成：

---
name: code-reviewer-v1
description: コードレビューをする時に使う（基本版）
tools: Read, Grep, Glob
---

コードをレビューしてください。
問題点を教えてください。
```

```
code-reviewer-v1 で exercises/level4/src/auth.py をレビューしてください
```

**出力を評価してください:**
- セキュリティ問題を全て発見しましたか？
- 報告は構造化されていますか？
- 優先度はついていますか？

## Step 4-3: 改善版レビューエージェントを作成する（v2）

v1 の問題点を踏まえて、プロンプトを改善します。

```
.claude/agents/code-reviewer-v2.md を作成：

---
name: code-reviewer-v2
description: セキュリティとパフォーマンス観点でコードレビューをする時に使う（改善版）
tools: Read, Grep, Glob
---

あなたはシニアソフトウェアエンジニアで、セキュリティとパフォーマンスの
専門家です。以下のフレームワークでコードレビューを行います。

## レビューフレームワーク

### フェーズ1: ファイル概要把握
- ファイルの目的と責務を特定する
- 依存関係とインポートを確認する

### フェーズ2: セキュリティ審査（OWASP Top 10 基準）
チェックリスト：
- [ ] インジェクション攻撃（SQL, コマンド, LDAP）
- [ ] 認証・セッション管理の問題
- [ ] 機密情報のハードコーディング
- [ ] 不適切なログ出力（パスワード等の露出）
- [ ] 入力バリデーション不足

### フェーズ3: パフォーマンス審査
チェックリスト：
- [ ] N+1 クエリパターン
- [ ] 不必要なメモリ使用（大きなデータの一括読み込み等）
- [ ] タイムアウトなしのネットワーク呼び出し
- [ ] 非効率なデータ構造・アルゴリズム

## 出力形式

### サマリー
| 重大度 | 件数 |
|--------|------|
| 🔴 Critical | X |
| 🟠 High | X |
| 🟡 Medium | X |
| 🟢 Low | X |

### 詳細

#### [重大度] 問題タイトル
**場所:** `ファイル名:行番号`
**説明:** 問題の技術的説明
**リスク:** 悪用された場合の影響
**修正方法:**
```python
# 修正後のコード例
```

### 総評
コードの全体的な品質評価（5点満点）と改善のロードマップ。
```

```
code-reviewer-v2 で exercises/level4/src/ を全ファイルレビューしてください
```

## Step 4-5: PR レビューエージェントを作成する

実際の開発フローに組み込めるエージェントを作ります。

```
.claude/agents/pr-reviewer.md を作成：

---
name: pr-reviewer
description: Pull Request のレビューをする時、変更差分を確認してレビューコメントを書く時に使う
tools: Read, Grep, Glob, Bash
---

あなたは Pull Request レビュアーです。
git diff の出力や変更ファイルを分析して、
GitHub の PR コメントとして使える形式でレビューを提供します。

## PR レビュー観点
1. 変更の意図と実装の整合性
2. テストカバレッジ
3. 破壊的変更の有無
4. セキュリティへの影響
5. パフォーマンスへの影響

## 出力形式
GitHub PR コメントとしてそのままコピペできる Markdown 形式で出力する。
承認/変更要求/コメントの判断も示す。
```

---

## チャレンジ課題

`code-reviewer-v2` を拡張して、以下も対応させてください:
1. Python 以外の言語（JavaScript/TypeScript）も適切にレビューできる
2. `.gitignore` にあるファイルはスキップする
3. テストファイル（`test_*.py`, `*.test.ts`）は別の観点でレビューする

- v1→v2 の改善で、同じツールセットでも劇的に品質が変わる
