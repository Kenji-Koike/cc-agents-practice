# Level 5: マルチエージェント連携

## 目標

- 複数のエージェントを組み合わせたワークフローを設計する
- エージェント間でのタスク分担を理解する
- 並列実行と逐次実行の使い分けを習得する

---

## Step 5-1: マルチエージェントの考え方

単一エージェントではなく複数を組み合わせる利点:

```
単一エージェント:
  [コード解析] → [セキュリティ] → [パフォーマンス] → [ドキュメント]
  ← 逐次実行、コンテキストが混在

マルチエージェント:
  [コード解析エージェント] ─┐
  [セキュリティエージェント] ─┤→ [統合レポートエージェント]
  [パフォーマンスエージェント] ─┘
  ← 並列実行可能、各エージェントが専門特化
```

**使い分けの原則:**
- 独立したタスク → 並列実行（速い）
- 前のタスクの結果が必要 → 逐次実行

---

## Step 5-2: 専門エージェント群を作成する

4つのエージェントを作成します。

### エージェント1: コードメトリクス収集

```
.claude/agents/metrics-collector.md を作成：

---
name: metrics-collector
description: コードのメトリクス（行数、複雑度、関数数など）を収集・計測する時に使う
tools: Read, Grep, Glob, Bash
---

あなたはコードメトリクス収集の専門家です。
指定されたコードベースから定量的なメトリクスを収集します。

## 収集するメトリクス
- 総ファイル数・総行数
- 関数・クラスの数
- 平均関数長（行数）
- コメント率（コメント行 / 総行数）
- TODO/FIXME/HACK コメントの数と場所

## 出力形式
JSON 形式でメトリクスを出力する（後続エージェントが処理しやすいように）

```json
{
  "files": [...],
  "summary": {
    "total_files": X,
    "total_lines": X,
    ...
  }
}
```
```

### エージェント2: 依存関係アナライザー

```
.claude/agents/dependency-analyzer.md を作成：

---
name: dependency-analyzer
description: コードの依存関係、インポート構造、外部ライブラリを分析する時に使う
tools: Read, Grep, Glob
---

あなたは依存関係分析の専門家です。

## 分析内容
- インポートされているモジュール・ライブラリの一覧
- 循環依存の検出
- 外部依存（標準ライブラリ以外）の特定
- 使われていないインポートの検出

## 出力形式
依存関係グラフをテキスト形式で表現し、問題点をリストアップする。
```

### エージェント3: テスト品質チェッカー

```
.claude/agents/test-checker.md を作成：

---
name: test-checker
description: テストコードの品質、カバレッジの推定、テスト漏れを確認する時に使う
tools: Read, Grep, Glob
---

あなたはテスト品質の専門家です。

## チェック内容
- テストファイルの存在確認
- テストされていない関数・クラスの特定
- エッジケースのテスト漏れ
- テストコードの品質（アサーションの適切さ、テスト名の明確さ）
- モックの過剰使用

## 出力形式
カバレッジ推定値（%）と未テスト関数の一覧を含む報告。
```

### エージェント4: 統合レポーター

```
.claude/agents/report-integrator.md を作成：

---
name: report-integrator
description: 複数の分析結果を統合して最終レポートを生成する時に使う
tools: Read
---

あなたは複数の分析レポートを統合して、
エグゼクティブサマリーを作成する専門家です。

入力として複数の分析結果（メトリクス、依存関係、テスト品質、セキュリティ）
を受け取り、以下の統合レポートを生成します：

## 統合レポート形式

### 総合ヘルススコア
各カテゴリを 0-100 でスコアリングし、加重平均で総合スコアを算出。

| カテゴリ | スコア | 重み |
|----------|--------|------|
| セキュリティ | ? | 40% |
| コード品質 | ? | 30% |
| テスト品質 | ? | 20% |
| 保守性 | ? | 10% |
| **総合** | **?** | |

### 優先アクションリスト
スコアが低いカテゴリから順に、具体的な改善アクションを3つ挙げる。

### 改善ロードマップ
- 今週対応すべきこと（Critical）
- 今月中に対応すべきこと（High）
- 次のスプリントで対応すべきこと（Medium）
```

## Step 5-3: マルチエージェントワークフローを実行する

以下のプロンプトで、複数エージェントを連携させてみましょう。

```
exercises/level4/src/ に対して以下の分析を並列で実行してください：
1. metrics-collector でコードメトリクスを収集
2. dependency-analyzer で依存関係を分析  
3. test-checker でテスト品質を確認

上記3つが完了したら、その結果を使って report-integrator で
統合レポートを生成してください。
```

**確認ポイント:**
- 3つのエージェントが並列に実行されましたか？
- それぞれの出力が最後の統合に使われましたか？
- 逐次・並列の実行パターンが正しく使われましたか？

## Step 5-4: 会話フック付きエージェントを作成する

エージェントが別のエージェントを呼び出すパターンを試します。

```
.claude/agents/code-health-checker.md を作成：

---
name: code-health-checker
description: コードの総合的な健全性チェックを実行する時に使う（オーケストレーター）
tools: Read, Grep, Glob
---

あなたはコード品質のオーケストレーターです。
指定されたコードベースに対して、以下の手順で総合チェックを行います：

1. まず全ファイルを把握する（Glob/Grep を使用）
2. コードの概要を理解する
3. 発見した問題点を重大度順に整理する
4. 改善のロードマップを提案する

## 重要な制約
- ファイルを変更しない（読み取りのみ）
- 発見した問題は必ず証拠（ファイル名:行番号）を示す
- 推測ではなく、実際のコードを根拠にする
```

```
code-health-checker を使って exercises/level3/sample/ の健全性チェックをしてください
```

## Step 5-5: エージェントのデバッグと改善

エージェントが期待通りに動かない場合のデバッグ方法:

```
# エージェントの動作を詳細に確認する
code-health-checker エージェントに exercises/level4/src/ を渡して、
どんなツールを呼び出すか観察しながら実行してください
```

**改善の観点:**
- エージェントが余計なことをしていないか？
- ツールの呼び出し順序は効率的か？
- 出力形式は使いやすいか？

## 最終チャレンジ: 自分だけのエージェントを作る

以下のいずれかを作成してください：

### オプション A: Git ヒストリー解析エージェント
- `git log`, `git blame` を使って変更履歴を分析
- よく変更されるファイル（不安定な箇所）を特定
- コントリビューター別の変更傾向を分析

### オプション B: ドキュメント生成エージェント
- ソースコードを読んで API ドキュメントを自動生成
- 関数シグネチャ、型、使用例を含む
- Markdown 形式で出力

### オプション C: リファクタリング提案エージェント
- コードの重複を検出
- 抽象化できるパターンを特定
- 具体的なリファクタリング手順を提案

---

## まとめ

- 複数の専門エージェントを組み合わせると、複雑なワークフローを構築できる
- 独立したタスクは並列実行、依存関係があるタスクは逐次実行
- エージェントの `description` を適切に書くと自動選択が機能する
- プロンプトの品質がエージェント出力の品質を決める
